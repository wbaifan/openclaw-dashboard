## 1. 项目概述

### 1.1 当前状态

OpenClaw Dashboard 是一个为 OpenClaw Gateway 提供实时监控的 Web 应用,采用赛博朋克风格设计。

**已实现功能**:
- ✅ Token 使用统计(30天趋势图)
- ✅ 今日统计(实时数据 + 小时活动热力图)
- ✅ 成本分解可视化
- ✅ 活跃会话监控
- ✅ 任务日志自动提取
- ✅ 实时活动流
- ✅ 多 Agent 活动聚合
- ✅ WebSocket 实时推送
- ✅ 设备身份认证(ed25519)

**部署状态**:
- 生产环境已部署,运行在 `127.0.0.1:3210`
- 通过 systemd 服务管理
- 已配置反向代理支持

### 1.2 技术栈

**前端**:
- React 19.2.4
- TypeScript 5.9.3
- Vite 8.0.0
- react-markdown 10.1.0

**后端**:
- Express 5.2.1
- WebSocket (ws 8.19.0)
- TypeScript 5.9.3
- chokidar 5.0.0 (文件监听)

**构建工具**:
- esbuild 0.27.4 (服务端打包)
- Vite (前端打包)

**项目结构**:
```
packages/
  server/    — Express + WebSocket 后端
  web/       — React 前端
dist/        — 统一构建输出
```

## 2. 已发现的问题

### 🔴 高优先级

#### P0-1: 缺少测试覆盖
**问题**: 项目完全没有测试代码  
**影响**: 
- 代码重构风险高
- 回归问题难以发现
- 代码质量无法保证

**具体表现**:
- 查找 `*.test.*` 和 `*.spec.*` 文件结果为空
- 没有单元测试、集成测试或 E2E 测试
- 关键逻辑(如 usage 聚合、活动追踪)未测试

**建议**: 
- 添加 Jest/Vitest 测试框架
- 优先覆盖核心业务逻辑(metrics.ts, activity-tracker.ts)
- 目标覆盖率: 70%+

---

#### P0-2: 多 Agent Usage 聚合未实现
**问题**: Token 使用量统计只支持单 Agent  
**影响**: 
- 多 Agent 场景下数据不准确
- 用户无法看到全局 token 使用情况

**具体表现**:
- `metrics.ts` 中 `loadUsageFromAllAgents()` 已实现扫描所有 Agent
- 但 `usageCost` 数据结构未明确区分单 Agent vs 多 Agent
- 前端显示未区分数据来源

**建议**: 
- 参考 `MULTI_AGENT_USAGE.md` 中的方案
- 实现多 Agent usage 聚合
- 前端显示全局总量 + 分 Agent 明细

---

#### P0-3: 错误处理不完善
**问题**: 错误处理较简单,缺少详细日志和用户提示  
**影响**: 
- 问题排查困难
- 用户体验差

**具体表现**:
```typescript
// server.ts 中的错误处理
process.on('uncaughtException', (err) => console.error('[fatal]', err.message));
process.on('unhandledRejection', (err) => console.error('[rejection]', (err as Error)?.message ?? err));

// WebSocket 连接失败后只打印日志,用户无感知
ws.onerror = () => ws.close();
```

**建议**: 
- 添加结构化日志(如 pino、winston)
- 前端显示友好的错误提示
- 后端记录详细错误堆栈
- 添加健康检查接口 `/health`

---

### 🟡 中优先级

#### P1-1: 性能优化空间
**问题**: 大数据量下可能存在性能瓶颈  
**影响**: 
- 响应速度下降
- 内存占用可能过高

**具体表现**:
```typescript
// metrics.ts - 扫描所有文件,可能很慢
const allFiles: string[] = [];
for (const { sessionsDir } of agentDirs) {
  const files = fs.readdirSync(sessionsDir)...
  allFiles.push(...)
}
await Promise.all(scanPromises); // 并发扫描所有文件

// activity-tracker.ts - 缓存 100 个文件的历史记录
for (const { filePath } of recentFiles.slice(0, 100)) {
  this._loadRecentFromFile(filePath);
}
```

**建议**: 
- 实现增量更新,只处理变更的文件
- 添加性能监控指标
- 限制并发扫描数量
- 考虑虚拟滚动(前端)

---

#### P1-2: 缺少加载状态和骨架屏
**问题**: 数据加载时界面空白,用户体验差  
**影响**: 
- 用户感觉应用卡顿
- 无法判断加载状态

**具体表现**:
```typescript
// useMetrics.ts - 初始状态为 null
const [data, setData] = useState<DashboardMetrics | null>(null);

// App.tsx - 直接使用可能为 null 的数据
const { data, wsStatus } = useMetrics();
const sessions = data?.status?.sessions?.recent ?? [];
```

**建议**: 
- 添加骨架屏组件
- 显示加载进度
- 添加数据为空的友好提示

---

#### P1-3: WebSocket 重连策略简单
**问题**: WebSocket 断开后重连逻辑简单,缺少指数退避  
**影响**: 
- 网络不稳定时频繁重连
- 服务器压力增大

**具体表现**:
```typescript
// useMetrics.ts - 固定 3 秒重连
ws.onclose = () => {
  setWsStatus('offline');
  reconnectRef.current = setTimeout(connect, 3000);
};
```

**建议**: 
- 实现指数退避重连(1s → 2s → 4s → 8s → 最大 30s)
- 添加心跳检测
- 显示重连状态

---

#### P1-4: 硬编码配置较多
**问题**: 关键参数硬编码,难以调整  
**影响**: 
- 修改参数需要改代码
- 不同环境难以适配

**具体表现**:
```typescript
// server.ts
const UPDATE_INTERVAL_MS = 10000;
const STARTUP_DELAY_MS = 3000;
const DEBOUNCE_MS = 500;

// activity-tracker.ts
const MAX_RECENT_ACTIVITY = 100;
const HISTORY_LOOKBACK_MS = 24 * 3600 * 1000;
const TAIL_READ_BYTES = 64 * 1024;
```

**建议**: 
- 将配置参数提取到配置文件或环境变量
- 支持运行时配置
- 添加配置文档

---

### 🟢 低优先级

#### P2-1: 类型定义可以更严格
**问题**: 部分类型使用 `unknown`,缺少精确类型  
**影响**: 
- 类型检查不够严格
- 潜在的运行时错误

**具体表现**:
```typescript
// metrics.ts
export interface DashboardMetrics {
  health?: unknown;  // 应该定义具体类型
  status?: unknown;  // 应该定义具体类型
  presence?: unknown; // 应该定义具体类型
}
```

**建议**: 
- 为 Gateway API 响应定义完整类型
- 使用 Zod 或 io-ts 进行运行时类型校验

---

#### P2-2: 缺少代码注释和文档
**问题**: 部分复杂逻辑缺少注释  
**影响**: 
- 代码理解困难
- 维护成本高

**具体表现**:
- `activity-tracker.ts` 中的任务提取逻辑复杂但注释少
- `metrics.ts` 中的 usage 聚合算法需要文档说明

**建议**: 
- 添加关键算法的注释
- 生成 API 文档
- 添加架构图

---

#### P2-3: 前端状态管理可以优化
**问题**: 使用 useState 管理复杂状态,可以考虑状态管理库  
**影响**: 
- 状态逻辑分散
- 代码可维护性一般

**建议**: 
- 评估是否需要状态管理库(Zustand/Jotai)
- 或者使用 React Context 优化状态传递

---

#### P2-4: 缺少性能监控
**问题**: 没有性能指标收集和分析  
**影响**: 
- 无法量化优化效果
- 性能问题难以发现

**建议**: 
- 添加前端性能监控(Web Vitals)
- 添加后端性能监控(响应时间、内存)
- 集成 APM 工具
